# LurkAway v1.1.0 — Phase 2 Implementation Plan

> **For Hermes:** Use the subagent-driven-development skill to implement this plan task-by-task.

**Version:** 1.1.0 (feature improvement, **non-breaking** — every new capability is opt-in and off by default except configurable timings, which default to the current hard-coded values).

**Goal:** Extend the shipped v1.0.0 anti-theft watchdog with three additions from the original plan's *Next Steps After v1*:

1. **Configurable timings** — expose walk-away delay, lid sensitivity, and siren volume in Settings (pure-local, low risk).
2. **Local evidence capture** — on tamper, grab one intruder snapshot + optional geotag, saved to disk only.
3. **Opt-in remote push** — off by default: on tamper, send an alert (with the snapshot + geotag) to the owner via **ntfy.sh**.

**Privacy stance (unchanged default):** v1.0.0 is on-device, no network, camera off while armed. This release preserves that as the **default**. Evidence capture is local-only. Remote push and location tagging are **explicit opt-in** and clearly labelled as leaving the device.

**Tech Stack (additions):** CoreLocation (one-shot geotag), URLSession (ntfy POST), Security/Keychain (optional ntfy token). No third-party dependencies.

**Target:** unchanged — macOS 26.5+, Apple Silicon + Touch ID.

---

## Architecture delta

Current pipeline (`LurkAway/AppState.swift`):

```
FaceDetectionManager ──onWalkAway──▶ arm() ──▶ MotionMonitor(power+lid)
                                                    │
                                          onMotion──▶ triggerAlarm() ──▶ AlarmController + LockScreenManager
```

v1.1.0 inserts an evidence + notification side-effect into `triggerAlarm()`, and makes three existing thresholds settings-driven:

```
triggerAlarm(trigger)
  ├─ armedOverlay.hide(); motionMonitor.stop()
  ├─ alarm.play(volume: settings.sirenVolume)      ← NEW: volume from settings
  ├─ presentLockScreen()
  └─ Task.detached { EvidenceReporter.capture(trigger) }   ← NEW, never blocks the siren
         ├─ EvidenceCaptureManager.snapshot()  → JPEG (only if settings.captureEvidence)
         ├─ LocationTagger.currentFix()        → CLLocation? (only if settings.tagLocation)
         ├─ persist JPEG + metadata to Application Support/LurkAway/Evidence/
         └─ NotificationSender.send(...)       → ntfy POST (only if settings.pushEnabled)
```

**Design constraints carried from v1:**
- Camera is **off** while armed. Evidence capture spins the camera up **only at alarm time**, grabs a single frame, and tears the session down immediately — mirroring `FaceDetectionManager`'s session lifecycle.
- The siren and lock overlay must fire **without waiting** on capture/network. Evidence work runs in a detached task; any failure is logged and swallowed (never blocks unlock).

### New files

```
LurkAway/Managers/
├─ EvidenceCaptureManager.swift   # one-shot front-camera JPEG grab
├─ LocationTagger.swift           # one-shot CoreLocation fix
├─ EvidenceReporter.swift         # orchestrates capture → persist → push
├─ NotificationSender.swift       # ntfy.sh HTTP client
└─ KeychainStore.swift            # stores the optional ntfy access token
LurkAway/Views/
└─ (SettingsView.swift gains "Detection" + "Alerts" tabs)
```

### Settings additions (`LurkAway/SettingsStorage.swift`)

| Key | Type | Default | Notes |
|-----|------|---------|-------|
| `walkAwayDelaySeconds` | `Double` | `5.0` | Maps to `noFaceThreshold = max(1, ceil(seconds / 1.5))` |
| `lidSensitivityDegrees` | `Double` | `5.0` | Feeds `LidAngleMonitor.thresholdDegrees` (current constant) |
| `sirenVolume` | `Double` | `1.0` | 0.1…1.0; drives siren amplitude + audible floor |
| `captureEvidence` | `Bool` | `true` | Local-only snapshot on tamper |
| `tagLocation` | `Bool` | `false` | Opt-in; requires location permission |
| `pushEnabled` | `Bool` | `false` | Opt-in; enables ntfy send |
| `ntfyServer` | `String` | `"https://ntfy.sh"` | Base URL |
| `ntfyTopic` | `String` | `""` | Shared secret — treat as sensitive (see Security) |

The ntfy **access token** (optional) is stored in the Keychain via `KeychainStore`, **not** UserDefaults (per Swift security rules).

## Task 4: EvidenceCaptureManager — one-shot snapshot

**Objective:** Capture a single front-camera JPEG on demand, powering the camera up only for the grab. **Optimize for the smallest file that still clearly shows the intruder's face** — this is evidence, not a photo album, and a small file also makes the Task 7 push fast.

**Files:**
- Create: `LurkAway/Managers/EvidenceCaptureManager.swift`

**Design:** Model the session lifecycle on `FaceDetectionManager` (`AVCaptureSession`, `.builtInWideAngleCamera`, serial queue). Use `AVCaptureVideoDataOutput`; **let auto-exposure/white-balance settle** by discarding the first few frames, capture the next frame, **downscale + JPEG-compress**, then `stopRunning()` and tear down inputs/outputs. Expose:

```swift
@MainActor
final class EvidenceCaptureManager {
    /// Powers the camera briefly, returns one compact JPEG frame, then releases the camera.
    /// Returns nil if the camera is unauthorized/unavailable. Times out after ~2.5s.
    func snapshot() async -> Data?
}
```

**Efficiency tuning (constants):**

| Constant | Value | Rationale |
|----------|-------|-----------|
| `sessionPreset` | `.hd1280x720` | Sharp source frame; downscaled after, so not the final size. Avoid `.photo` (heavy) and the `.cif352x288` used for detection (too small to ID a face). |
| `settleFrames` | `3` | Discard the first 3 delivered frames so auto-exposure/white-balance converge — biggest readability win. Still well inside the timeout. |
| `targetLongEdge` | `800` px | Downscale so the longest edge is 800px. Enough for a clearly readable full-frame face; a fraction of 720p's pixels. |
| `jpegQuality` | `0.5` | Aggressive compression; faces stay recognizable. Roughly targets a 30–80 KB file. |
| color | keep | Colour aids identification (clothing, hair); grayscale saves little more. |
| metadata | strip | Encode from the downscaled bitmap only — no EXIF; smaller and no incidental data. |

**Pipeline:** `CVPixelBuffer` → `CIImage`/`CGImage` → downscale to `targetLongEdge` (aspect-preserving, e.g. `CGContext` or `NSImage` draw) → `NSBitmapImageRep.representation(using: .jpeg, properties: [.compressionFactor: jpegQuality])`. Do the downscale+encode off the main actor (on the capture/util queue).

- Guard `AVCaptureDevice.authorizationStatus(for: .video) == .authorized`; if not authorized, return `nil` (do **not** prompt during an alarm).
- Track a delegate frame counter; ignore until `settleFrames` reached, then resume a `CheckedContinuation` with the encoded JPEG. Resume exactly once; a ~2.5s timeout resumes `nil` if frames never arrive.
- Throttle the video output to a low FPS (reuse `CameraTuning.throttle`) so we don't waste frames while settling.

**Concurrency:** follow the existing `nonisolated(unsafe)` + `@MainActor` hop pattern in `FaceDetectionManager` for delegate callbacks; keep strict-concurrency clean.

**Verification:** Add a temporary debug hook that calls `snapshot()` and writes to `/tmp/lurk-test.jpg`. Confirm: (a) valid JPEG, face clearly readable; (b) file size in the ~30–80 KB range (`ls -l`); (c) the frame is well-exposed (not the dark cold-start frame); (d) camera indicator light turns **off** immediately after. Remove the debug hook before commit.

**Note (future):** HEIC would roughly halve the size at equal quality but adds encoder complexity and an ntfy `Content-Type` concern — deferred, not in v1.1.0.

**Step 2: LocationTagger.**

```swift
@MainActor
final class LocationTagger: NSObject, CLLocationManagerDelegate {
    /// Prompts for authorization if undecided. Call when the user enables location tagging
    /// in Settings — never during an alarm.
    func requestAuthorization()
    /// One fix, or nil on denial/timeout (~4s). Never prompts.

```

- Use `requestLocation()` (single delivery) bridged to a `CheckedContinuation` with a 4s timeout. Delegate methods are `nonisolated` + `MainActor.assumeIsolated` (callbacks arrive on the main run loop).

## Task 6: EvidenceReporter — persist snapshot + metadata

**Objective:** Orchestrate capture + geotag and write evidence to disk. Local-only path complete before push is added.

**Files:**
- Create: `LurkAway/Managers/EvidenceStore.swift`
- Create: `LurkAway/Managers/EvidenceReporter.swift`
- Modify: `LurkAway/AppState.swift`

**Step 1: EvidenceStore (filesystem + lifecycle).** Small, `nonisolated` filesystem helper so the alarm path and the Settings UI share one owner of the evidence folder. This is also where **deletion** lives — the user must be able to wipe stored photos.

```swift
enum EvidenceStore {
    /// ~/Library/Application Support/LurkAway/Evidence
    static var directoryURL: URL { get }

    /// Writes <ISO8601>.jpg (+ sidecar .json metadata). Returns the JPEG URL or nil on failure.
    @discardableResult
    static func save(jpeg: Data?, metadata: Metadata) -> URL?

    /// Number of stored evidence images (for the Settings label / disabling the clear button).
    static func itemCount() -> Int

    /// Deletes every stored image + sidecar. Used by the "Clear all evidence" button.
    static func clearAll() throws
}
```

- `directoryURL` resolved via `FileManager.url(for: .applicationSupportDirectory, …)`; create with `withIntermediateDirectories: true`.
- Filenames: ISO8601 (colon-free, filesystem-safe) so images sort chronologically.
- Sidecar JSON `Metadata`: `trigger`, `date`, optional `latitude`/`longitude`.
- All calls wrap I/O; `save` swallows+logs errors (never throws into the alarm), while `clearAll` throws so the UI can surface a failure.
- Log the saved path `.private`; never log coordinates `.public`.

**Step 2: EvidenceReporter (orchestration).**

```swift
@MainActor
final class EvidenceReporter {
    struct Evidence { let jpegURL: URL?; let jpeg: Data?; let location: CLLocation?; let trigger: AlarmTrigger; let date: Date }

    private let camera = EvidenceCaptureManager()
    private let location = LocationTagger()

    /// Runs capture + geotag concurrently, persists via EvidenceStore, returns the record.
    func capture(trigger: AlarmTrigger, settings: SettingsStorage) async -> Evidence
}
```

- Run snapshot and location **concurrently** with `async let` (gate each on its setting).
- Persist via `EvidenceStore.save`; keep the in-memory JPEG in the returned `Evidence` for the Task 7 push (avoid re-reading from disk).
- All I/O wrapped so failure is logged, not thrown into the alarm path.

**Step 3: Wire into AppState.triggerAlarm().** After `presentLockScreen()`:

```swift
if settings.captureEvidence || settings.tagLocation || settings.pushEnabled {
    let reporter = evidenceReporter
    let snapshot = settings // captured value
    Task.detached(priority: .utility) {
        let record = await reporter.capture(trigger: trigger, settings: snapshot)
        await reporter.deliver(record, settings: snapshot)   // Task 7 fills deliver()
    }
}
```

Hold a `private let evidenceReporter = EvidenceReporter()` on `AppState`. The detached task must never touch alarm/unlock state.

**Verification:** Trigger a tamper (unplug AC while armed). Siren fires immediately (no delay). Within ~2–3s a JPEG + JSON appear under Application Support/LurkAway/Evidence. Unlock still instant.

## Task 7: NotificationSender — ntfy client (dual mode)

**Objective:** Send an opt-in tamper alert via ntfy. **Two delivery modes off one request**, no server setup, no account:
- **Topic push (default) — full text + photo.** Subscribers of the topic (ntfy app, or the topic URL opened in a browser) get the intruder photo inline plus the message/maps link.
- **Email (optional, add-on) — text only.** If the user also enters an email, ntfy forwards a plain email (title + message + maps link; the photo is not inline in email, but stays saved locally). Receiver installs nothing.

Both are the same `POST {server}/{topic}`; email is just an extra header. Default enables the topic (photo) mode.

**Files:**
- Create: `LurkAway/Managers/NotificationSender.swift`
- Create: `LurkAway/Managers/KeychainStore.swift`
- Modify: `LurkAway/Managers/EvidenceReporter.swift` (add `deliver`)

**ntfy contract (HTTP):** `POST {server}/{topic}`. When a JPEG is present it is the raw body (photo notification for topic subscribers). Headers:
- `Title: LurkAway tamper — <trigger>`
- `Priority: urgent`
- `Tags: rotating_light`
- `Filename: <ISO8601>.jpg` — marks the body as an attachment (topic-photo mode).
- `Message`/`X-Message` — human summary; append a maps link when a fix exists: `https://maps.apple.com/?ll=<lat>,<lon>`.
- `Email: <addr>` — **only when the user set an email**; ntfy also forwards a text email (no inline photo).
- `Authorization: Bearer <token>` — only if a token is stored (Keychain; needed for auth'd/self-hosted servers, not for anonymous ntfy.sh).

If there is no JPEG, POST the text in the body (still delivers to topic + email).

```swift
struct NotificationSender {
    struct Alert { let title: String; let message: String; let jpeg: Data?; let filename: String; let email: String? }
    /// Returns true on 2xx. Never throws into the caller; logs failures.
    func send(_ alert: Alert, server: String, topic: String, token: String?) async -> Bool
}
```

- Validate `URL(string:)` for `server`/`topic`; bail (log) on malformed input — never force-unwrap.
- `Message`/`Title`/`Email` header values must be header-safe: reject/strip CR/LF to prevent header injection; only include `Email` when it looks like a valid address.
- 10s `URLSession` timeout; single attempt (best-effort — the siren is the primary deterrent; no retry outbox in v1.1.0).

**KeychainStore:** minimal `get/set/delete(account:)` wrapper over `SecItem*` for the optional ntfy token. No token in UserDefaults.

**EvidenceReporter.deliver():** if `settings.pushEnabled`, build the `Alert` from the record (message summarizes trigger + local time; adds maps link when a fix exists; `email = settings.ntfyEmail` when non-empty) and call `send(...)`, reading the token from `KeychainStore`.

**Settings addition:** add `ntfyEmail: String` (default `""`) to `SettingsStorage` (UserDefaults; an email address is not a secret).

**Verification:** Set a personal ntfy topic. (a) Subscribe via the ntfy app **or** just open `https://ntfy.sh/<topic>` in a browser → trigger tamper → intruder photo + maps link arrive inline. (b) Also enter an email → trigger tamper → a plain email arrives with the text + maps link, no app needed. With push off (default), no network call is made — verify with `nettop` that the app opens no connection.

## Task 7b: ntfy setup & troubleshooting guide

**Objective:** Make the feature genuinely easy to adopt with an in-repo guide and an in-app link to it.

**Files:**
- Create: `docs/ntfy-setup.md`
- Modify: `LurkAway/Views/SettingsView.swift` (a "How to set up…" link in the Alerts tab, Task 8)
- Modify: `README.md` (Task 11 links to the guide)

**`docs/ntfy-setup.md` covers:**
1. **What ntfy is** — a free push service; no account or server needed for the hosted ntfy.sh.
2. **Pick a topic** — use the in-app "Generate" button (long random string) so nobody else can guess it. Explain the topic is effectively a shared secret.
3. **Receive the photo (topic mode):** open `https://ntfy.sh/<topic>` in a browser and allow notifications, **or** install the ntfy app (iOS/Android/desktop) and subscribe to the topic.
4. **Receive by email (optional):** enter your email in Settings; alerts arrive as a normal email (text + maps link; photo stays on the Mac).
5. **Test:** use the in-app "Send test" button; expect a notification/email within seconds.
6. **Troubleshooting:** nothing arrives → check topic spelling matches on both ends; browser didn't ask to allow notifications → re-open the URL and allow; email in spam; corporate network blocking ntfy.sh (try the app on cellular); self-hosted server → set the server URL and, if it requires auth, the access token; rate limits on anonymous email.
7. **Privacy note:** enabling remote alerts sends the notification (and, in topic mode, the photo) to the configured ntfy server over HTTPS.
8. **Costs & limits:** LurkAway itself is free and runs no servers — it just posts to whatever ntfy server you set. The hosted **ntfy.sh free tier covers normal use** (tamper alerts are rare; the photo is ~30–80 KB, well under the 15 MB / 100 MB attachment limits; attachments expire after 3h on the server but are delivered immediately and also saved on your Mac). **Email forwarding is the most rate-limited** free path (a few/day) — use topic push as primary. For unlimited volume or full privacy, **self-host ntfy** (free) and put its URL in the server field; heavy users can pay for ntfy Pro. No change needed in LurkAway either way.

**Verification:** The Alerts tab "How to set up…" link opens the guide (GitHub URL or a local Help sheet). Guide steps reproduce a working end-to-end alert for a new user.

## Security & privacy considerations

- **Intruder photo = personal data.** Capture is local-only by default; remote send is opt-in and labelled. Document that recording a person may carry legal obligations in some jurisdictions.
- **ntfy topic is a shared secret.** Anyone who knows `{server}/{topic}` can read the alerts (and photos). Mitigate: the "Generate" button creates a long random topic; support an optional access token (stored in **Keychain**, never UserDefaults); all traffic is HTTPS. Do not log the topic or token.
- **Camera-off invariant** now has one bounded exception (single frame at alarm). Ensure the session is always torn down, even on error/timeout, so the indicator light cannot stay on.
- **No new attack surface while idle/armed:** network is only touched inside the detached post-alarm task, and only when `pushEnabled`.
- **Input validation:** validate `server`/`topic` with `URL(string:)`; never force-unwrap user-entered URLs.
- **Failure isolation:** capture/geotag/network run in a detached utility task; nothing there can delay the siren, the lock overlay, or Touch ID unlock.

## Risks & open questions

| Risk | Mitigation |
|------|-----------|
| Camera won't power up fast enough at alarm for a usable frame | 2s timeout + first-frame grab; photo is best-effort, siren is primary |
| ntfy.sh third-party dependency/availability | User-configurable server (self-hostable); send is best-effort, non-blocking |
| Location prompt appearing mid-alarm is jarring | Request authorization the first time `tagLocation` is toggled on in Settings, not during the alarm; skip if not authorized |
| Photo capture conflicts with face-detection camera use | They never run together — face detection stops at `arm()`; capture only at alarm; both fully release the device |
| Strict-concurrency violations in new async camera/location code | Reuse the proven `nonisolated(unsafe)` + `@MainActor` hop pattern from `FaceDetectionManager` |
| Users expect photo but denied camera permission | Alerts tab shows camera authorization status; caption explains capture needs camera access |
